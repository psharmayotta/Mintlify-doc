# Q0 API Reference

The Q0 Inference API gives you programmatic access to every model in the Q0 platform — the same models available in the Playground, Marketplace, and Deployments — via a simple, consistent HTTP interface. Use it to build production applications, integrate Q0 models into your backend services, or automate batch inference workflows.

<Note>
  All confirmed endpoint URLs, header names, payload fields, and default values in this reference are sourced directly from Q0 UI screenshots (Playground → View Code panel and Deployments → API Documentation section).
</Note>

---

## Base URL

All API requests are made to the Q0 inference gateway:

```text
https://adminapi-alpha.q0.new/inference/api/inference/api-key
```

<Tip>
  This endpoint is the same one generated by the **View Code** button in the Q0 Playground. It handles all model types — Text Generation, Text-to-Speech, Audio-to-Text, Image Generation, and Vision-Language models — with the model determined by the `model-id` header on each request.
</Tip>

---

## Authentication

Every request to the Q0 API requires two authentication headers:

| Header | Required | Description |
| --- | --- | --- |
| `api-key` | ✅ Yes | Your Q0 JWT API token. Generate and manage tokens in **Settings → API Keys**. Tokens are shown as `eyJh••••OcLE` in the dashboard; reveal the full value using the eye icon. |
| `model-id` | ✅ Yes | The unique identifier of the model you want to invoke. Copy the model ID from the model's entry in **My Models** (the short hash shown below the display name, e.g. `U2FsdGVkX18MuUVQj2NdeYC1tzQ4qKZ8Yv3M2oZkux4=`). |
| `content-type` | ✅ Yes | Always `application/json`. |

<Warning>
  **Never expose your `api-key` in client-side code, browser console logs, or public repositories.** API keys are JWT tokens that grant full access to your Q0 account quota and billing. Store them server-side as environment variables or in the Q0 Secrets vault.
</Warning>

### Getting your API key

<Steps>
  <Step title="Go to Settings → API Keys">
    Navigate to [Settings → API Keys](https://ui-alpha.q0.dev/settings) in the Q0 dashboard.
  </Step>
  <Step title="Generate a new key">
    Click **Generate API Key** (top-right). Give the key a descriptive name (e.g. `prod-inference-key`).
  </Step>
  <Step title="Copy the full token immediately">
    The full JWT value is only shown once. Copy it and store it securely — subsequent views show the masked form (`eyJh••••...`).
  </Step>
  <Step title="Check expiry">
    Each key shows an **Expires On** field (e.g. `in 14 days`). Rotate keys before expiry to avoid service interruption. Keys with status **Expired** will return `401 Unauthorized`.
  </Step>
</Steps>

## Request Format

All requests use `HTTP POST` with a JSON body.

### Request headers

```http
POST https://adminapi-alpha.q0.new/inference/api/inference/api-key
Content-Type: application/json
api-key: <your-jwt-api-key>
model-id: <your-model-id>
```

### Request body

```json
{
  "output_tokens": 4096,
  "temperature": 0.7,
  "system_prompt": "Act as a helpful assistant to the user",
  "top_p": 0.95,
  "text_input": "Your prompt goes here",
  "frequency_penalty": 0.5,
  "presence_penalty": 0.5
}
```

## Request Parameters

All parameters below are confirmed from the Q0 Playground configuration panel and the Deployment API Documentation section.

### Core parameters

<ParamField body="text_input" type="string" required>
  The user's prompt or input text. This is the main message sent to the model — equivalent to the user turn in a conversation, or the query in a RAG pipeline.
</ParamField>

<ParamField body="output_tokens" default="4096" type="integer" required>
  Maximum number of tokens the model will generate in its response. Higher values allow longer outputs but increase cost and latency.

  | Model category | Recommended range |
  | --- | --- |
  | Short Q&A / classification | 64 – 512 |
  | Summarisation / analysis | 512 – 2048 |
  | Long-form generation | 2048 – 8192 |

  **Slider range in Playground:** 64 – 8192
</ParamField>

<ParamField body="system_prompt" default="Act as a helpful assistant to the user" type="string" required>
  A standing instruction that defines the model's role, tone, and behaviour for the entire session. Applied before the user's `text_input` on every request.

  ```json
  "system_prompt": "You are a customer support agent for an Indian fintech company. Respond politely and concisely in English."
  ```
</ParamField>

<ParamField body="temperature" default="0.7" type="float" required>
  Controls randomness in the model's output.

  - `0.0` → fully deterministic, always picks the highest-probability next token
  - `0.7` → balanced creativity and coherence (Q0 default)
  - `2.0` → maximum randomness

  **Slider range:** 0 – 2

  <Tip>
    For factual retrieval, classification, or structured data extraction, use `temperature: 0.1` or lower. For creative writing or brainstorming, use `0.8` – `1.2`.
  </Tip>
</ParamField>

<ParamField body="top_p" default="0.95" type="float" required>
  Nucleus sampling threshold. The model samples only from the smallest set of next-token candidates whose cumulative probability is ≥ `top_p`.

  - `1.0` → considers all tokens (no filtering)
  - `0.95` → Q0 default — filters out unlikely tail tokens while preserving diversity
  - `0.1` → very focused; only the most probable tokens are considered

  **Slider range:** 0 – 1
</ParamField>

### Repetition control parameters

<ParamField body="frequency_penalty" default="0.5" type="float">
  Reduces the likelihood of the model repeating tokens that have already appeared frequently in the output. Positive values penalise repetition more strongly.

  **Slider range:** -2 to 2 | **Q0 default:** 0.5
</ParamField>

<ParamField body="presence_penalty" default="0.5" type="float">
  Encourages the model to introduce new topics and tokens it hasn't used yet in the current response, rather than elaborating on existing content.

  **Slider range:** -2 to 2 | **Q0 default:** 0.5

  <Note>
    **Frequency Penalty vs Presence Penalty:** Frequency Penalty scales with how many times a token has appeared (the more it appears, the more it is penalised). Presence Penalty applies a flat penalty to any token that has appeared at all, regardless of frequency. Use both together to strongly discourage repetition in long-form outputs.
  </Note>
</ParamField>

### Batch Inference additional parameters

When creating a Batch Inference job via the Q0 dashboard, two additional parameters are available that are not used in the standard per-request API:

<ParamField body="top_k" default="3" type="integer">
  Limits the model to sampling from only the top K most probable next tokens at each generation step. Lower values produce more focused outputs.

  **Slider range (Batch Inference):** 3 – 7
</ParamField>

<ParamField body="score_threshold" default="0" type="float">
  Minimum quality score for a Batch Inference response to be included in the output dataset. Responses scoring below this threshold are filtered out.

  **Slider range:** 0 – 1 | Set to `0` to include all responses regardless of quality.
</ParamField>

## Code Examples

All examples below use the confirmed Q0 endpoint and header structure from the Playground View Code panel.

<CodeGroup>

```python Python
import requests
import json

url = "https://adminapi-alpha.q0.new/inference/api/inference/api-key"

payload = {
    "output_tokens": 4096,
    "temperature": 0.7,
    "system_prompt": "Act as a helpful assistant to the user",
    "top_p": 0.95,
    "text_input": "Explain the key benefits of RAG over pure LLM inference",
    "frequency_penalty": 0.5,
    "presence_penalty": 0.5
}

headers = {
    "api-key": "YOUR_Q0_API_KEY",
    "model-id": "YOUR_MODEL_ID",
    "content-type": "application/json"
}

response = requests.post(url, headers=headers, json=payload)
print(response.status_code)
print(response.text)
```

```java Java
import java.net.URI;
import java.net.http.HttpClient;
import java.net.http.HttpRequest;
import java.net.http.HttpResponse;

String url = "https://adminapi-alpha.q0.new/inference/api/inference/api-key";

String payload = """
    {
      "output_tokens": 4096,
      "temperature": 0.7,
      "system_prompt": "Act as a helpful assistant to the user",
      "top_p": 0.95,
      "text_input": "Explain the key benefits of RAG over pure LLM inference",
      "frequency_penalty": 0.5,
      "presence_penalty": 0.5
    }
    """;

HttpClient client = HttpClient.newHttpClient();
HttpRequest request = HttpRequest.newBuilder()
    .uri(URI.create(url))
    .header("api-key", "YOUR_Q0_API_KEY")
    .header("model-id", "YOUR_MODEL_ID")
    .header("content-type", "application/json")
    .POST(HttpRequest.BodyPublishers.ofString(payload))
    .build();

HttpResponse<String> response = client.send(request, HttpResponse.BodyHandlers.ofString());
System.out.println(response.body());
```

## API Keys — Management Reference

Managed from **Settings → API Keys** (`ui-alpha.q0.dev/settings`).

### API Keys table columns

| Column | Description |
| --- | --- |
| **API Key Name** | The display name you gave the key when generating it (e.g. `Test`, `TOKEN 1780573388238`) |
| **API Keys** | Masked JWT value (`eyJh••••OcLE`). Click the eye icon to reveal the full token — only shown once at generation time. |
| **Expires On** | Time until expiry (e.g. `in 14 days`) or time since expiry (`20 days ago`) |
| **Creation Date** | Date the key was created (e.g. `11 Jun 2026`) |
| **Status** | `Valid` (green) — key is active and accepted by the API. `Expired` (red) — key will return 401 on all requests. |

### Current Playground Token

A special token is shown at the top-right of the API Keys tab: **Current Playground Token** (`eyJh••••OcLE` with an eye icon). This is the active token used by the Q0 Playground and is automatically managed — you do not need to rotate it manually. Use it for quick testing; for production integrations, generate a dedicated key with an appropriate expiry.

## Deployment API (Private Endpoint)

When you deploy a model via **Deployments → Create Deployment**, Q0 provisions a private GPU endpoint for that model. The API Documentation section of every deployment detail page provides ready-to-use code snippets for that specific endpoint.

### Private endpoint request format

The payload structure for a private deployment endpoint is identical to the shared inference API, with the endpoint URL replaced by your deployment's specific URL (visible in the deployment detail page once status is `Healthy`):

```python
import requests
import json

# Replace with your deployment's endpoint URL (from Deployments → Detail → Model Endpoint)
url = "YOUR_DEPLOYMENT_ENDPOINT_URL"

payload = {
    "output_tokens": 4096,
    "temperature": 0.7,
    "top_p": 0.95,
    "system_prompt": "Act as a helpful assistant to the user",
    "text_input": "Your prompt here",
    "frequency_penalty": 0.5,
    "presence_penalty": 0.5
}

headers = {
    "api-key": "YOUR_Q0_API_KEY",
    "model-id": "YOUR_MODEL_ID",
    "content-type": "application/json"
}

response = requests.post(url, headers=headers, json=payload)
print(response.json())
```

### Deployment auto-scaling parameters (for reference)

The following fields are configured in the Deployments wizard (Step 4 — Scaling) and affect how your private endpoint handles concurrent load. They are not part of the inference request body but are visible in the deployment right panel:

| Parameter | Default | Description |
| --- | --- | --- |
| Enable Auto Scaling | On | Automatically scales replica count based on thresholds below |
| Min Replicas | 0 | Minimum pods — `0` enables scale-to-zero (cold start on first request) |
| Max Replicas | 2 | Maximum pods under peak load |
| Scale Up Window | 0s | How quickly Q0 scales up when load increases |
| Scale Down Window | — | How long Q0 waits before scaling down after load drops |
| CPU Usage | — | % threshold to trigger scale-up |
| GPU Usage | — | % threshold to trigger scale-up |
| GPU RAM | — | % threshold to trigger scale-up |
| Concurrency | — | Active Requests Per Pod threshold to trigger scale-up |
| Latency | — | Millisecond P99 threshold to trigger scale-up |

## HTTP Status Codes

| Status | Meaning | What to do |
| --- | --- | --- |
| `200 OK` | Request succeeded | Parse the response body |
| `400 Bad Request` | Malformed request body or invalid parameter value | Check your JSON payload against the parameter reference above |
| `401 Unauthorized` | Missing or invalid `api-key` header, or key is expired | Check key status in Settings → API Keys. Regenerate if expired. |
| `403 Forbidden` | Valid key but not authorised to access this model or resource | Verify the `model-id` belongs to your organisation's My Models |
| `429 Too Many Requests` | TPM or RPM limit exceeded for this model | Implement exponential backoff. Check quota in Settings → Quota. |
| `500 Internal Server Error` | Q0 platform error | Retry with exponential backoff. If persistent, check platform status. |
| `503 Service Unavailable` | Deployment not yet ready (Health Check: Unhealthy) | Wait for deployment status to reach `Healthy` before sending requests |

## Secrets & Credential Management

When integrating Q0 models into external services (cloud storage, databases, custom model registries), credentials must be stored as **Secrets** in the Q0 platform rather than hardcoded in application code.

Navigate to **Integrations → Secrets** in the left sidebar to create and manage secrets. Secrets are referenced by name across My Models, Datasets, Batch Inference, and Knowledge Base — never exposing the raw credential value.

### Supported secret providers

| Provider | Used in | Credential type |
| --- | --- | --- |
| AWS | My Models, Datasets, Batch Inference, Knowledge Base | Access key ID \+ Secret access key |
| GCP | My Models, Datasets, Batch Inference, Knowledge Base | Service account JSON key |
| Hugging Face | My Models, Training | HF access token |
| Azure | My Models, Datasets, Knowledge Base | Connection string or service principal |
| Docker Hub | My Models (Docker source, Private registry) | Username \+ access token |

### Using a secret in an API integration

When configuring a Batch Inference job, Dataset source, or Knowledge Base source programmatically via the Q0 dashboard, select **Existing Secret** and choose the secret name from the dropdown. The secret value is injected at job runtime — it never appears in your request payload or application logs.
