## What you're building

An "agent" on Q0 is a loop, not a model. The model — served by your **LLM Inference Engine** deployment — proposes actions, your application executes them against real systems, and the results feed back into the conversation until the task is done.

This guide walks through a **Deployment Ops Agent**: an assistant that can check the health of a Q0 deployment, pull recent benchmark numbers, and draft a status update for a stand-up — all through tool calls, with no fine-tuning required.

The model never touches your infrastructure directly — it only ever asks the agent loop to run a named function and hands back a plain-text result. That boundary is what makes tool calling safe to run against production endpoints.

## Prerequisites

| Requirement | Notes |
| --- | --- |
| A deployed model | Any **LLM** category deployment works; pick a reasoning-capable checkpoint if the agent needs multi-step planning |
| Endpoint type | **Shared** is fine for development; move to **dedicated** once the agent is called on a schedule or by multiple users |
| Region | Navi Mumbai (NM1) or Delhi (DL1), whichever is closer to where your tools/data live, to keep tool round-trips fast |
| API key | Generated from your Q0 account settings |

<Note>
  Quantization affects tool-calling reliability. F16 and AWQ checkpoints tend to follow tool schemas more consistently than aggressively compressed formats — worth testing before you commit to a quantization for an agent that runs unattended.
</Note>

## Step 1 — Define what the agent is allowed to do

Every tool is a contract: a name, a description the model uses to decide _when_ to call it, and a strict parameter schema. Keep the surface area small on purpose — three well-scoped tools beat ten vague ones.

```python
tools = [
    {
        "type": "function",
        "function": {
            "name": "get_deployment_status",
            "description": "Return live health, GPU tier, and region for a Q0 deployment endpoint.",
            "parameters": {
                "type": "object",
                "properties": {
                    "endpoint_id": {"type": "string", "description": "Q0 deployment endpoint ID"}
                },
                "required": ["endpoint_id"],
            },
        },
    },
    {
        "type": "function",
        "function": {
            "name": "get_benchmark_summary",
            "description": "Return the latest latency and throughput benchmark run for an endpoint.",
            "parameters": {
                "type": "object",
                "properties": {
                    "endpoint_id": {"type": "string"}
                },
                "required": ["endpoint_id"],
            },
        },
    },
    {
        "type": "function",
        "function": {
            "name": "draft_standup_note",
            "description": "Compose a short stand-up update summarizing deployment health and any incidents.",
            "parameters": {
                "type": "object",
                "properties": {
                    "findings": {"type": "string", "description": "Free-text notes to summarize"}
                },
                "required": ["findings"],
            },
        },
    },
]
```

## Step 2 — Implement the tools against real Q0 data

The functions themselves are ordinary code — they can call the Q0 platform APIs, your internal monitoring, or a database. The model never sees this implementation, only the return value.

```python
import requests

Q0_BASE_URL = "https://api.q0.dev/v1"
HEADERS = {"Authorization": f"Bearer {Q0_API_KEY}"}

def get_deployment_status(endpoint_id: str) -> str:
    r = requests.get(f"{Q0_BASE_URL}/deployments/{endpoint_id}", headers=HEADERS)
    d = r.json()
    return f"{d['name']}: {d['status']}, {d['gpu_tier']} in {d['region']}"

def get_benchmark_summary(endpoint_id: str) -> str:
    r = requests.get(f"{Q0_BASE_URL}/deployments/{endpoint_id}/benchmarks/latest", headers=HEADERS)
    b = r.json()
    return f"p50 latency {b['p50_ms']}ms, throughput {b['tokens_per_sec']} tok/s"

def draft_standup_note(findings: str) -> str:
    return f"Deployment health — {findings}"

available_tools = {
    "get_deployment_status": get_deployment_status,
    "get_benchmark_summary": get_benchmark_summary,
    "draft_standup_note": draft_standup_note,
}
```

## Step 3 — Run the agent loop

The loop is the same four moves every time: send messages \+ tools, check for a tool call, execute it locally, append the result, and call the model again. It ends when the model responds without asking for another tool.

```python
from together import Together  # Q0's LLM Inference Engine speaks an OpenAI-compatible chat API

client = Together(base_url=Q0_BASE_URL, api_key=Q0_API_KEY)

def run_agent(user_message: str, model: str, max_turns: int = 6):
    messages = [
        {"role": "system", "content": "You manage Q0 deployments. Use tools before answering."},
        {"role": "user", "content": user_message},
    ]

    for _ in range(max_turns):
        response = client.chat.completions.create(
            model=model, messages=messages, tools=tools, tool_choice="auto"
        )
        msg = response.choices[0].message
        messages.append(msg.model_dump())

        if not msg.tool_calls:
            return msg.content  # done — final answer

        for call in msg.tool_calls:
            fn = available_tools[call.function.name]
            args = json.loads(call.function.arguments)
            result = fn(**args)
            messages.append(
                {"role": "tool", "tool_call_id": call.id, "content": result}
            )

    return "Agent did not converge within max_turns."
```

```python
print(run_agent(
    "Check endpoint ep-mumbai-llm-01 and write a stand-up note.",
    model="your-deployed-model-name",
))
```

## Moving from prototype to production

- **Cap `max_turns`.** Agents that loop on tool calls without a ceiling can spin indefinitely on ambiguous instructions — treat it like a timeout.
- **Log every tool call and result.** This is your audit trail if the agent ever takes an unexpected action against a deployment.
- **Promote to a dedicated endpoint** once the agent runs on a schedule, so its latency doesn't compete with interactive traffic on a shared endpoint.
- **Re-benchmark after switching quantization.** A format that saves memory can shift how reliably the model emits well-formed tool calls.
