Compare AI models or hardware configurations to measure performance, accuracy, and efficiency on Q0

The **Benchmarking** module lets users systematically evaluate and compare AI models — or the hardware they run on — using consistent datasets and metrics. Instead of relying on published leaderboard numbers or anecdotal testing in the Playground, Benchmarking gives users a repeatable, configurable test run on Q0's own infrastructure, so decisions about which model (or which GPU) to use in production are backed by real, side-by-side data. 

This module is designed for users who need to answer questions like-

-  Which of these two models gives better quality for my use case at acceptable cost? 
- Does this model perform differently on H100 vs L40S GPUs? or 
- How well does my RAG pipeline perform against my own knowledge base?

**What Benchmarking Helps Users Do-**

- **Compare Models Objectively**: Run two (or more) models — from different sources, providers, or checkpoints — against the same dataset and metrics to see how they really stack up.

- **Compare Hardware Configurations**: Evaluate inference performance (latency, throughput, cost) of the same model across different GPU types and counts, to choose the most cost-effective deployment setup.
- **Evaluate a Single Model**: Run standardized NLP benchmarks against one model to understand its baseline capabilities on specific tasks.
- **Validate RAG/Knowledge Base Pipelines**: Test how well a Knowledge Base pipeline retrieves and generates answers, before relying on it in production.
- **Make Data-Driven Decisions**: Get quantified, side-by-side comparison results (covered in detail in a later section) instead of guessing which model or hardware setup is "good enough."
- **Reuse and Track Runs**: Every benchmarking run is saved with its configuration, status, and results, so users can revisit, compare across runs, and avoid re-running identical tests.

The main Benchmarking page lists every benchmarking run created by the user or team, along with its type, status, and key details.

#### Step 1 — Basic Details

Add Benchmarking Name and Description to identify this benchmarking experiment

#### Step 2 — Benchmarking Type

Select “**Model Benchmarking”,** Compares performance between two different AI models — using the same or different model sources — across selected metrics like accuracy, latency, and cost.

**Model Category:** The type of model to benchmark, based on task or modality (e.g., `Diffusion Model` for text-to-image, or text/vision/audio models). This determines which model classes and configuration fields appear later.

<Frame>

**Configuring Model 1 and Model 2**

·       **Model Source:** Where the model comes from — e.g., `Huggingface` (external repository) or a cloud provider such as `GCP` (internal/cloud-hosted model). |

·       **Model Path:** The repository path or storage location where the model is located. For Huggingface, this is the repo path (e.g., `moonshotai/K****struct`); for cloud sources like GCP, it's typically a storage URI (e.g., `gs://inferen****truct/`). Q0 verifies the path automatically, shown with a green **Verified** badge once confirmed.

·       **Model Class**: The specific model architecture/pipeline class used to load and run the model (e.g., `StableDiffusion3Pipeline`). This must match the model's actual implementation for the benchmark to run correctly.

<Frame>
  ![Image](/images/image-47.png)
</Frame>

·       **Secret**: Authentication credentials required to access the model source. Choose between: **Existing Secret** — select a previously saved credential from the dropdown (e.g., a saved Huggingface token or GCP service account). Create **New Secret** — add a new credential on the fly if one doesn't already exist.

·       **Select Secret**: Dropdown to pick the specific saved secret to use (e.g., `Hugg`, `GCPtest`), scoped to the chosen Model Source.

<Tip>
  **How to select Model Source, Path, and Class**: 

  First choose the **Model Source** (where the model lives — Huggingface, GCP, etc.). This determines the expected format of **Model Path** (a repo ID for Huggingface, a storage URI for cloud providers). Then select the matching **Model Class** that corresponds to the model's architecture/pipeline. Finally, attach a **Secret** so Q0 can authenticate and pull the model — Q0 verifies the path automatically and marks it **Verified** once access is confirmed. 

  This same Model Source / Model Path / Model Class / Secret block is repeated for **Model 2**, allowing two independently-sourced models (even from different providers, as shown with Model 1 on Huggingface and Model 2 on GCP) to be benchmarked against each other.
</Tip>

#### **Step 4 — Inferencing Settings**

This step configures how each model generates output during the benchmark run. Metrics are automatically enabled based on the benchmarking goal, and settings are configured independently for **Model 1** and **Model 2 **

<Frame>
  ![Image](/images/image-49.png)
</Frame>

·       **No of Images**: (For image/diffusion models) The number of images each model should generate per dataset prompt during the benchmark, adjustable via slider or input (e.g., `3`, within a `1–4` range).

·       **Show Advanced Options:** Expands additional generation parameters per model, including:

o   T**emperature:** Controls randomness of output. Higher values make outputs more creative; lower values make them more deterministic and consistent.

o   **Top\_p:**  Limits token selection to a probability mass, balancing output diversity against coherence.

o   **Max Tokens:** Sets the maximum length of the generated output.

o   **Stop Sequences:** Defines specific tokens or phrases at which the model should stop generating further output.

o   **Streaming:** Enables real-time, token-by-token output generation instead of waiting for the full response.

o   **Seed:** A fixed value that ensures reproducible outputs across multiple runs, useful for fair, repeatable comparisons.

<Note>
  These advanced settings mirror the same generation controls available in the Playground (see \[Playground\](/playground) documentation) but applied consistently across the entire benchmark dataset for each model.
</Note>

·       **Verifying Model:** Q0 confirms that both models (and their credentials/paths) are valid and accessible.

## **Hardware to Hardware Benchmarking**

**Hardware Benchmarking** compares how the same model performs across two different hardware setups — different GPU types or different GPU counts — under identical workloads. This is the right benchmarking type when the question is not _which model_ is better, but _which infrastructure_ is most performant or cost-efficient for a model you've already chosen.

#### **Step 1 — Basic Details**

Same as Model Benchmarking. Provide a unique **Benchmarking Name** (e.g., `Bench_Hardware`) and an optional **Description** (up to 500 characters), then click Continue.

#### **Step 2 — Benchmarking Type: Hardware Configuration**

Select ‘**Hardware Benchmarking**’ as the type. This reveals two independent hardware configuration blocks — **Hardware 1** and **Hardware 2** — to define the two setups being compared.

<Frame>
  ![Image](/images/image-52.png)
</Frame>

 Both blocks follow the same structure: 

- **Hardware (GPU Type)**: Select the GPU model for this configuration. Options shown with specs and hourly cost: 

       • **H100** — 8 CPUs, 80.00GB HBM3, (high-end, highest throughput)

       • **L40S** — 4 CPUs, 48.00GB GDDR6 ECC, (cost-efficient, mid-range)

- **GPU Count Per Node**: Number of GPUs to allocate per node for this hardware setup. 

For example: Hardware 1 with H100 × 3 GPUs vs. Hardware 2 with L40S × 2 GPUs.

- **Concurrent Users**: The number of simultaneous users/requests to simulate against each hardware setup during the benchmark.

 Tests how each configuration handles load. Available values: `1`, `5`, `10`, `15`, `20`. Multiple values can be selected to test each hardware setup across several load levels in a single run — for example, `1` tests single-user latency while `10` or `20` tests throughput under realistic concurrent load.

<Card title="Concurrent Users" type="note">
  **Concurrent Users** is the key variable for Hardware Benchmarking. A setup that excels at low concurrency (1 user) may fall behind at higher load (10–20 users). Selecting multiple concurrency levels gives you a performance curve for each hardware configuration across the full range of expected traffic, not just a single data point.
</Card>

#### **Step 3 — Model Configuration**

Specify the single model that will be deployed on both hardware setups for comparison. The fields here are the same as in Model Benchmarking — Model Source, Model Path, Model Class, and Secret — but only one model is configured, since the variable being compared is the hardware, not the model. See \[Configuring Model 1 and Model 2\](#configuring-model-1-and-model-2) for field-level details.

<Frame>
  ![Image](/images/image-53.png)
</Frame>

#### **Step 4 — Dataset**

Select the dataset the model will run against on both hardware configurations. The same dataset options are available as in Model Benchmarking. For LLM/text model hardware benchmarking, standard NLP benchmarks from the Dataset Library are commonly used:

 

<Frame>
  ![Image](/images/image-54.png)
</Frame>

**MMLU**: Massive Multitask Language Understanding — a broad academic benchmark spanning 57 subjects including math, science, law, and history.

**HellaSwag:** A commonsense reasoning benchmark that tests how well a model can complete a sentence or scenario description in context.  

Multiple datasets can be selected simultaneously (e.g., both `MMLU` and `HellaSwag`) so each hardware configuration is evaluated across more than one task type in a single run.

#### **Step 5 — Inferencing Settings**

Defines how the selected model generates responses during the benchmark. Since hardware — not the model — is the variable being tested, these settings are configured once and applied identically on both hardware setups to ensure a fair comparison. The model being benchmarked is shown at the top of this step (e.g., `Llama3-1-8B`).

 

<Frame>
  ![Image](/images/image-55.png)
</Frame>

 

·       **Max Tokens**`4096` (range: 64–8192) | Maximum tokens the model can generate per response. Longer outputs stress hardware memory and compute more, making this a key variable in throughput tests.

·       **Temperature:** `0.7` (range: 0–2) | Controls output randomness. Lower values produce deterministic responses; higher values introduce variety.

·       **System Prompt:** `"Act as a helpful assistant to the user"` | Sets the model's role for all prompts in the dataset. Kept identical across both hardware setups.

·       **Top K:** `3` (range: 3–7) | Limits sampling to the top-K most probable next tokens at each step.

·       **Score Threshold:** `0` (range: 0–1) | Minimum relevance/confidence score for outputs to be included in metric calculations.

 

Clicking **Show Advanced Options** reveals additional generation controls: 

·       **Top-P**: `0.95` (range: 0–1) | Nucleus sampling — limits token candidates to a cumulative probability ≥ Top-P, balancing coherence and diversity.

·       **Frequency Penalty**`0.5` (range: -2 to 2) | Reduces repetition of tokens already used frequently in the output.

·       **Presence Penalty**`0.5` (range: -2 to 2) | Encourages the model to introduce new topics/tokens rather than repeating what's already been said.

 

After Step 5, a full configuration summary is displayed covering all settings for both hardware configurations. Review it, then click **Create Benchmarking** to launch the run. The benchmark then progresses through the same status pipeline as all other benchmarking types.

## **Model Evaluation**

**Model Evaluation** assesses a single model's capability on a specific task using standard, widely recognized benchmarks. Unlike Model Benchmarking (which compares two models against each other), Model Evaluation is about understanding _how well a single model performs in absolute terms_ — for example, how accurately it answers factual questions, how faithfully it transcribes speech, or how realistically it generates images. Results are scored against established metrics (accuracy, BLEU, WER, FID, etc.) so teams can verify that a model meets the bar required for their use case before deploying it.

**What Model Evaluation Helps Users Do**

 

·       **Validate baseline quality** of a model before committing to a deployment or fine-tuning investment.

·       **Verify task-specific capability** using standardized tasks (Question Answering, Summarization, OCR, Speech Recognition, etc.) rather than informal testing.

·       **Generate auditable evidence** of model performance for compliance, governance, or client-facing reporting purposes.

·       **Identify weaknesses** in a model across specific sub-tasks, so fine-tuning or model selection decisions are well-informed.

·       **Compare against industry benchmarks** by running the same evaluation tasks used in published research and leaderboards.

Result Navigation Tabs

**1. Task Metrics**

Task Metrics measure how accurately and correctly the model completed the evaluation task against the selected dataset. These are the primary quality indicators for the chosen Evaluation Task (e.g., Question Answering, Visual QA, Summarization).

| Metric | Description | Ideal Value |
| --- | --- | --- |
| \*\*F1\*\* | The harmonic mean of Precision and Recall. Measures how well the model's answers overlap with the reference answers — balancing false positives (incorrect content generated) and false negatives (correct content missed). A score of \`1.0\` is perfect; \`0\` means no overlap. | Higher is better (max 1.0) |
| \*\*AnIs\*\* | Average Normalized Inverse Score — a normalized measure of answer quality that accounts for variations in phrasing. Useful for tasks where partial credit should be given for near-correct answers. | Higher is better |
| \*\*Accuracy\*\* | The percentage of evaluation samples for which the model's output exactly or substantially matches the correct answer. Computed as: \`Correct predictions / Total predictions\`. | Higher is better (max 1.0 / 100%) |
| \*\*Exact Match\*\* | The strictest accuracy metric — the proportion of samples where the model's output is a character-for-character match with the reference answer, with no tolerance for minor phrasing differences. Used heavily in QA benchmarks like SQuAD and MMLU. | Higher is better (max 1.0) |
| \*\*Choice Accuracy\*\* | For multiple-choice tasks (e.g., MMLU), the percentage of questions where the model selected the correct answer option (A, B, C, D). More lenient than Exact Match — scores the answer choice, not the full text. | Higher is better (max 1.0) |
| \*\*Avg Latency Ms\*\* | The average time (in milliseconds) the model took to generate a response per sample across the entire evaluation dataset. Included in Task Metrics as a quick latency reference alongside quality scores. | Lower is better |

**2. Performance Metrics**

Performance Metrics measure the speed and throughput characteristics of the model under the evaluation workload — critical for understanding whether the model can meet latency SLAs in production.

| Metric | Unit | Description | Ideal Value |
| --- | --- | --- | --- |
| Avg Latency Ms | ms | The mean inference latency across all evaluation samples — average time from request submission to response completion. | Lower is better |
| Max Latency Ms | ms | The maximum observed latency across all samples — the worst-case response time seen during the evaluation run. | Lower is better |
| Min Latency Ms | ms | The minimum observed latency — the fastest response time seen across all samples. | Lower is better |
| P50 Latency Ms | ms | The 50th percentile (median) latency — half of all requests were completed faster than this value. A robust central-tendency measure less affected by extreme outliers than the mean. | Lower is better |
| \*\*P95 Latency Ms\*\* | ms | The 95th percentile latency — 95% of requests completed within this time. A key SLA metric for real-world applications where tail latency matters. | Lower is better |
| \*\*P99 Latency Ms\*\* | ms | The 99th percentile latency — 99% of requests completed within this time. Reflects worst-case performance for the vast majority of traffic; critical for latency-sensitive production workloads. | Lower is better |
| \*\*Latency Std Dev Ms\*\* | ms | Standard deviation of latency across all samples — measures how consistent (or variable) the model's response times are. A high standard deviation indicates unpredictable latency spikes. | Lower is better |
| \*\*Total Tokens Input\*\* | count | The total number of input tokens processed across all evaluation samples (e.g., \`1,200\`). Useful for cost estimation and understanding the total prompt volume the model handled. | Reference value |
| \*\*Avg Tokens Per \[Request\]\*\* | count | The average number of input tokens per evaluation sample — reflects the typical prompt length for this dataset and task. | Reference value |
| \*\*Total Tokens Generated\*\* | count | The total number of output tokens generated by the model across all samples (e.g., \`100\`). | Reference value |
| \*\*Avg Input Length Tokens\*\* | count | Average number of tokens per input prompt across the dataset (e.g., \`12\`). Indicates how concise or detailed the evaluation prompts were. | Reference value |
| \*\*Avg Output Length Tokens\*\* | count | Average number of tokens per model response across the dataset (e.g., \`1\`). For choice-based tasks like MMLU, this is typically very small (1–3 tokens). | Reference value |

**3. Reliability Metrics**

Reliability Metrics measure the stability and robustness of the model's responses across the full evaluation dataset — indicating how dependable the model is when handling a large volume of diverse inputs.

| Metric | Unit | Description | Ideal Value |
| --- | --- | --- | --- |
| \*\*Error Rate\*\* | % | The percentage of evaluation samples for which the model returned an error — failed requests, timeouts, or malformed responses — rather than a valid output (e.g., \`100.00%\` means all requests failed; \`0%\` means none failed). | Lower is better (target: 0%) |
| \*\*Timeout Rate\*\* | % | The percentage of requests that exceeded the configured response timeout threshold and were terminated before a response was produced. | Lower is better (target: 0%) |
| \*\*Response Rate\*\* | % | The percentage of evaluation samples for which the model returned any response (successful or otherwise). Complement of Error Rate: \`Response Rate \+ Error Rate = 100%\` when no timeouts occur. | Higher is better (target: 100%) |
| \*\*Total Samples\*\* | count | The total number of evaluation samples in the dataset that were submitted to the model (e.g., \`100\`). Confirms the full evaluation scope. | Reference value |
| \*\*Successful Samples\*\* | count | The number of samples for which the model returned a valid, parseable response (e.g., \`0\` in a failed run). | Higher is better (target: Total Samples) |
| \*\*Empty Response Rate\*\* | % | The percentage of requests for which the model returned an empty or null response — even when no error was technically raised (e.g., \`100.00%\` means all responses were empty). A high Empty Response Rate alongside a high Error Rate typically indicates an authentication, access, or model configuration issue. | Lower is better (target: 0%) |

<Note>
  **Interpreting Reliability Metrics**: An `Error Rate` of 100% combined with an `Empty Response Rate` of 100% and `Successful Samples` of 0 indicates that none of the model's requests completed successfully. This typically points to an infrastructure or authentication issue (e.g., restricted model access, invalid credentials, network failure) rather than a model quality issue. Check the benchmarking run's logs for specific error messages
</Note>

**4. Output Quality**

Output Quality metrics evaluate characteristics of the model's generated text beyond simple accuracy — measuring safety, diversity, consistency, and language richness of the responses across the full dataset.

| Metric | Unit | Description | Ideal Value |
| --- | --- | --- | --- |
| \*\*Toxicity Score\*\* | 0–1 | A measure of how much harmful, offensive, or inappropriate content appears in the model's outputs, scored by an automated toxicity classifier. A score of \`0\` means no toxic content was detected; \`1\` means all outputs were flagged as toxic. | Lower is better (target: 0) |
| \*\*Repetition Score\*\* | 0–1 | Measures how often the model repeats the same phrases, sentences, or token sequences within or across its responses. High repetition indicates degenerate output patterns (e.g., the model getting "stuck" repeating content). | Lower is better (target: 0) |
| \*\*Hallucination Rate\*\* | % | The estimated percentage of responses containing factually incorrect or fabricated information not grounded in the prompt or dataset context. Measured by automated fact-checking or consistency checking against the dataset's reference answers. | Lower is better (target: 0%) |
| \*\*Std Dev Output Length\*\* | tokens | Standard deviation of output length in tokens across all responses — measures consistency of response length. A high value means the model produces wildly variable-length outputs; a low value means responses are consistently sized. | Lower is better for consistency |
| \*\*Vocabulary Richness (TTR)\*\* | 0–1 | Type-Token Ratio — the ratio of unique tokens to total tokens across all generated responses. Higher TTR means the model uses a diverse, rich vocabulary; lower TTR indicates repetitive or formulaic language patterns. | Higher is better for creative/generative tasks; lower may be acceptable for structured QA |
| \*\*Avg Output Length Tokens\*\* | tokens | Average number of tokens across all model responses (e.g., \`1\`). A very low average (1–2 tokens) is expected for multiple-choice tasks; higher values are expected for summarization, generation, or reasoning tasks. | Task-dependent |
| \*\*Max Output Length Tokens | tokens | The longest response generated across all evaluation samples (e.g., \`1\`). Indicates the upper bound of model verbosity for this dataset and configuration. | Task-dependent |
| Min Output Length Tokens | tokens | The shortest response generated across all evaluation samples (e.g., \`1\`). Along with Max and Avg, forms the complete output length distribution. | Task-dependent |
