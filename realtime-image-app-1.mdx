## What you're building

"Real-time" for image generation doesn't mean instant — it means the gap between a user typing a prompt and seeing a result is short enough that the interaction feels like a live conversation rather than a batch job. That gap is mostly determined by two decisions you make at deployment time: which **GPU tier** you deploy on, and which **quantization format** the model runs in.

Under the hood this is a **Diffusion Model** category deployment — the same deployment mechanics as an LLM endpoint, but the request/response shape is prompt-in, image-out.

## Prerequisites

| Decision | Recommendation |
| --- | --- |
| GPU tier | **H100** for sub-few-second generations; **L40S** is more cost-effective for non-interactive/batch generation |
| Endpoint type | **Dedicated** — image generation is compute-heavy per request, so a warm dedicated instance avoids cold-start latency that a shared endpoint can introduce under load |
| Quantization | **FP8** or **MXFP4** meaningfully cut latency with limited quality loss; keep **F16** if visual fidelity matters more than speed |
| Region | Pick NM1 or DL1 based on where the majority of your users are, to minimize network round-trip |

## 1. Deploy the diffusion model

From the Q0 dashboard, create a deployment with model category **Diffusion Model**, choose your GPU tier and quantization, and select a dedicated endpoint for consistent latency.

```python
import requests

Q0_BASE_URL = "https://api.q0.dev/v1"
HEADERS = {"Authorization": f"Bearer {Q0_API_KEY}"}

deploy_resp = requests.post(
    f"{Q0_BASE_URL}/deployments",
    headers=HEADERS,
    json={
        "model_category": "diffusion",
        "model_name": "your-selected-diffusion-model",
        "endpoint_type": "dedicated",
        "gpu_tier": "H100",
        "quantization": "FP8",
        "region": "NM1",
    },
)
endpoint_id = deploy_resp.json()["endpoint_id"]
```

## 2. Submit a generation job

Image generation is asynchronous even on a dedicated endpoint — you submit a job and get a `job_id` back immediately, rather than blocking on the full generation.

```python
def submit_generation(endpoint_id: str, prompt: str, steps: int = 25, width: int = 1024, height: int = 1024):
    r = requests.post(
        f"{Q0_BASE_URL}/deployments/{endpoint_id}/generate",
        headers=HEADERS,
        json={
            "prompt": prompt,
            "steps": steps,
            "width": width,
            "height": height,
        },
    )
    return r.json()["job_id"]

job_id = submit_generation(endpoint_id, "a lighthouse at dusk, watercolor style")
```

Fewer `steps` generally means faster generation at some cost to detail — a useful dial to expose to users who want speed over fidelity.

## 3. Poll for the result

```python
import time

def wait_for_image(endpoint_id: str, job_id: str, timeout_s: int = 30):
    start = time.time()
    while time.time() - start < timeout_s:
        r = requests.get(
            f"{Q0_BASE_URL}/deployments/{endpoint_id}/jobs/{job_id}",
            headers=HEADERS,
        )
        data = r.json()
        if data["status"] == "complete":
            return data["image_url"]
        if data["status"] == "failed":
            raise RuntimeError(data.get("error", "generation failed"))
        time.sleep(0.5)
    raise TimeoutError("generation did not complete in time")

image_url = wait_for_image(endpoint_id, job_id)
```

## 4. Make the frontend feel live

Short-poll from the browser so the UI updates the moment the job completes, rather than waiting on a single long request:

```tsx
async function generateAndDisplay(prompt: string, setImageUrl: (url: string) => void) {
  const { job_id } = await fetch("/api/generate", {
    method: "POST",
    body: JSON.stringify({ prompt }),
  }).then((r) => r.json());

  const poll = setInterval(async () => {
    const status = await fetch(`/api/generate/${job_id}`).then((r) => r.json());
    if (status.status === "complete") {
      clearInterval(poll);
      setImageUrl(status.image_url);
    }
  }, 500);
}
```

For a smoother feel, show a progress indicator based on `steps` elapsed if your job status response includes step-level progress, rather than a generic spinner.

## 5. Tune for your actual latency budget

| Lever | Effect |
| --- | --- |
| GPU tier (H100 vs L40S) | Biggest single lever on raw generation speed |
| Quantization (F16 → FP8 → MXFP4) | Progressively faster, progressively more compression artifacts — test visually before committing |
| `steps` | Linear tradeoff between speed and detail |
| Resolution | Higher width/height scales compute roughly quadratically |
| Endpoint type | Dedicated avoids queuing behind other tenants' jobs on a shared endpoint |

Use the **Benchmarking** module to measure actual p50/p95 generation time for your chosen combination under realistic concurrent load — the right settings for a demo rarely survive contact with real traffic unchanged.
