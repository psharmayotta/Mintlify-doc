#### Securely store and manage cloud credentials for use across Q0 workflows

####  

**Secrets** is Q0's secure credential management module. It lets users store cloud provider authentication keys and tokens in one place — and reference them by name across any workflow that requires external access, such as importing datasets, connecting model sources for Benchmarking, Batch Inference, or Deployments. 

Instead of entering raw API keys or access credentials directly into each workflow form, users create a named Secret once and select it by name whenever credentials are needed. This keeps sensitive credentials off form fields, reduces the risk of accidental exposure, and makes credential rotation simple — update the Secret in one place and every workflow that references it automatically uses the updated credentials.

**Why Secrets Are Required**  

Every workflow on Q0 that connects to an external cloud provider — to pull a dataset, load a model, or push results — needs authentication credentials. Without a secure, centralized store for these credentials, users would have to paste raw API keys into multiple forms across Training, Benchmarking, Batch Inference, Deployments, Datasets, and Knowledge Base workflows. 

**Secrets solve three specific problems:** 

- **Security**: Credentials are stored encrypted and referenced by name, never exposed as plaintext in form fields or job configurations. 

- **Reusability**: A single Secret can be referenced across Datasets, Benchmarking runs, Batch Inference jobs, and Deployments without re-entering credentials each time. 

- **Maintainability**: When a key is rotated or expires, the user updates it in one Secret record — all workflows using that Secret automatically inherit the new credentials without reconfiguration. 

# **Where Secrets Are Used Across Q0**

- **Datasets**: Authenticate access to AWS S3, GCP Cloud Storage, Huggingface Hub, or Azure Blob Storage when importing dataset files. | 

- **Benchmarking**: Authenticate access to externally hosted models (Model Source) during Model Benchmarking and Hardware Benchmarking runs. 

- **Batch** **Inference**: Authenticate access to the model source (e.g., GCP-hosted model) during batch job setup.  

- **Deployments**: Authenticate access to custom model files stored on cloud infrastructure. 

- **Knowledge** **Base**: Authenticate database connections and cloud storage sources during Knowledge Base ingestion.

## **Step 2 — Setup**

Configure the secret data 

- **Secret Type:** The cloud provider or service this secret authenticates with. Selecting a type loads the correct JSON credential template for that provider. 

- **Secret Data**: The actual credentials entered in JSON format, following the template provided for the selected Secret Type.

## **Supported Secret Types** 

| **Secret Type ** | ** Provider ** | ** JSON Credential Template ** |
| :-- | :-- | :-- |
| AWS | Amazon Web Services | `{ "access_key_id": "", "secret_access_key": "" }` |
| GCP | Google Cloud Platform | Service account JSON key (includes `type`, `project_id`, `private_key_id`, `private_key`, `client_email`, etc.) |
| Huggingface | Huggingface Hub | API token / access token for authenticating with the Huggingface model and dataset registry. |
| Azure | Microsoft Azure | Azure storage account credentials (connection string or account name \+ key). |

The **Secret Data** field is a JSON editor that pre-fills with the credential template for the selected provider, so users only need to paste in their actual key values. The `JSON` label in the top-right corner of the editor indicates the expected format. 

<Warning>
  Never share your Secret Data with anyone or paste credentials into unsecured channels. Q0 stores Secret Data encrypted at rest. Always use least-privilege IAM roles or service accounts — grant only the permissions Q0 needs to access the specific bucket or resource, not full account access.
</Warning>

## **Secrets Dashboard** 

Once secrets are created, the **Secrets** list page provides a central view for managing all saved credentials.  

User can – 

- **Create Secret :** Launches the secret creation wizard to add a new credential. 

- **Search Secrets:** Search bar to filter the secret list by display name.

  ![Screenshot 2026 07 02 125606](/images/Screenshot-2026-07-02-125606.png)

| **Column ** | ** Description ** |
| :-- | :-- |
| Name | The display name alongside its unique system ID (truncated, with a copy icon). |
| Secret Type | The cloud provider the secret authenticates with, shown as a provider logo and label |
| Creation Details | The user who created the secret and the timestamp |
| Status | The module context where this secret has been used or validated, shown as a colored badge . This gives a quick signal of which workflows are actively using the credential. |
